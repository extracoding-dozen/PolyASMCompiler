# Спецификация языка PolyASM (Payload Language)

Данный документ описывает синтаксис, типы данных и встроенные функции специализированного языка для написания позиционно-независимых шеллкодов (standalone shellcode).

Главная философия языка: **Отсутствие внешних зависимостей (Zero Dependencies).** Код не линкуется с `libc`, не имеет секций `.data` или `.rodata`. Все операции транслируются напрямую в системные вызовы (syscalls) ядра Linux x86_64, а данные хранятся исключительно в регистрах и на стеке.

---

## 1. Типы данных и переменные

Создание переменных происходит простым присваиванием. Компилятор автоматически выделяет место на стеке или резервирует виртуальный регистр на этапе генерации IR.

*   `qword` — 64-битное целое число без знака (размер машинного слова).
*   `string` — строка символов.

### Особенности хранения строк
Так как шеллкод не может обращаться к абсолютным адресам секции `.rodata`, строки инициализируются динамически через стек. 
Например, `string path = "/bin/sh"` компилятор превратит в:
```nasm
mov rax, 0x0068732f6e69622f ; "/bin/sh\x00" в little-endian
push rax
mov rdi, rsp                ; rdi теперь содержит указатель на строку
```

**Пример:**
```pascal
string target_dir = "/tmp/.hidden"
qword delay = 15
```

## 2. Управление потоком выполнения

Язык поддерживает классические ветвления и циклы. Под капотом компилятор генерирует условные переходы (`cmp`, `je`, `jne`) и метки базовых блоков, которые затем проходят через `ControlFlowFlattener` для запутывания анализаторов.

```pascal
// Ветвления
if get_file_size("/etc/shadow") > 0 {
    copy("/etc/shadow", "/tmp/bak")
} else {
    exit(1)
}

// Циклы
qword i = 0
while i < 10 {
    sleep(1)
    i = i + 1
}
```

## 3. Базовые команды (Standard Library)

Большинство команд транслируются в 1-3 системных вызова Linux. Некоторые команды (например, `useradd`) являются "макросами" и разворачиваются в цепочку вызовов `execute`.

### ⚡ Выполнение и процессы
| Команда                    | Описание | Зачем нужна в payload'е |
|:---------------------------| :--- | :--- |
| `execute(path, arg1, ...)` | Обертка над `sys_execve`. Запускает бинарник с аргументами. | Базовая функция для запуска системных утилит, локеров, майнеров или дропперов. |
| `fork()`                   | Обертка над `sys_fork`. Создает копию процесса. | Критично для шеллкода. Позволяет пейлоаду уйти в фон, чтобы процесс, в который нас заинжектили, не упал. |
| `exit(code)`               | Вызов `sys_exit`. Корректное завершение потока. | Без нее процессор дойдет до конца шеллкода и начнет выполнять мусор (краш SIGSEGV). |
| `int sleep(seconds)`       | Вызов `sys_nanosleep`. Приостанавливает выполнение. | Anti-Sandbox техника. Позволяет переждать время анализа в песочнице антивируса. |

### 📁 Файловая система
| Команда                         | Описание | Зачем нужна в payload'е                                                                                                                               |
|:--------------------------------| :--- |:------------------------------------------------------------------------------------------------------------------------------------------------------|
| `int copy(src, dst)`            | Открывает оба файла и копирует байты (через `sys_sendfile` или цикл `read/write`). | Кража файлов, создание резервных копий (например, при ransomware) или клонирование бинарника в скрытую папку. 0 если копирование успешно, -1 - ошибка |
| `int rename(old, new)`          | Обертка над `sys_rename`. Перемещение/переименование файла. | Быстрое скрытие артефактов или подмена системных бинарников (состояние гонки, подмена после копирования). 0 - успех, -1 - ошибка                      |
| `int write(path, offset, data)` | Запись сырых байт (qword/string) по смещению в файл. | Инжект вредоносного кода в легитимные файлы, модификация `~/.bashrc` для закрепления.   0 - успех, -1 - ошибка                                                              |
| `int delete(path)`              | Обертка над `sys_unlink`. Удаление файла. | Зачистка следов (удаление исходного дроппера).  0 - успех, -1 - ошибка                                                                                                      |
| `int get_file_size(path)`       | Возвращает `qword` (вызов `sys_stat`). | Проверка успешности кражи файла или определение размера перед чтением/шифрованием.                                                                    |

### 🛡 Права и Пользователи
| Команда                   | Описание | Зачем нужна в payload'е |
|:--------------------------| :--- | :--- |
| `chmod(path, mode)`       | Изменяет права доступа (`sys_chmod`). | Выдача `+x` загруженному стейджу перед вызовом `execute`. |
| `chown(path, uid, gid)`   | Смена владельца файла (`sys_chown`). | Подготовка файлов для выполнения от имени нужного пользователя (например, при повышении привилегий). |
| `setattr(path, attr)`     | Установка файловых атрибутов (аналог `chattr +i`). | Защита бэкдора от удаления антивирусом или администратором (Immutable flag). |
| `int useradd(user, pass)` | **Макрокоманда.** Модифицирует `/etc/passwd` и `/etc/shadow` напрямую или вызывает `execute("/sbin/useradd")`. | Создание теневого пользователя (бэкдора) для постоянного доступа по SSH. 0 - успех, -1 - ошибка|
| `passwd(user, new_pass)`  | **Макрокоманда.** Смена пароля пользователя. | Блокировка доступа легитимному админу (Ransomware/Wiper логика). |

### 🌐 Сеть
| Команда | Описание | Зачем нужна в payload'е |
| :--- | :--- | :--- |
| `connect(ip, port)` | Устанавливает TCP-соединение и перенаправляет `stdin`, `stdout`, `stderr` (через `sys_dup2`) в сокет. | Создание **Reverse Shell**. Позволяет получить интерактивный контроль над машиной. |

## 4. Интеграция внешнего ASM (Foreign Function Interface)

Одной из главных фич компилятора является возможность импорта сторонних файлов `.s` (написанных на чистом ассемблере, Intel syntax). 

Это необходимо для внедрения проприетарной логики (например, кастомных криптоалгоритмов ChaCha20, специфичных LPE эксплойтов), которую трудно или нецелесообразно писать на высокоуровневом PolyASM.

### Механика работы
1. Директива `import` указывает компилятору распарсить сторонний `.s` файл.
2. Парсер находит экспортированные метки (функции) в файле.
3. Код из `.s` файла конвертируется во внутренний IR-формат компилятора.
4. **Важно:** Импортированный код **также проходит через `ObfuscatorEngine`**! Это значит, что сторонний, общеизвестный эксплойт будет автоматически зашифрован, его строки спрятаны, а граф потока выполнения изменен, что сделает его невидимым для сигнатурного анализа.

### Синтаксис

```pascal
// Импорт внешнего файла с ассемблерным кодом
import "modules/custom_crypto.s"

string data = "secret_payload"

// Вызов функции encrypt_payload, которая экспортирована из custom_crypto.s
// Аргументы передаются в соответствии с System V AMD64 ABI (rdi, rsi, rdx...)
qword result = call_asm("encrypt_payload", data, 14)

if result == 1 {
    write("/tmp/enc.dat", 0, data)
}
```
### Пример кода
```pascal
string log_file = "/tmp/hack.log"
string backup = "/tmp/hack.bak"
string hacker_name = "phantom"

qword size = get_file_size(log_file)

if size > 0 {
    write(log_file, size, "System compromised\n")
    rename(log_file, backup)
}

chmod("/tmp/hack.bak", 0777)
sleep(10)
exit(0)
```

**Требования к внешнему `.s` файлу:**
* Код должен быть позиционно-независимым (не использовать абсолютные адреса памяти).
* Функция должна соблюдать соглашение о вызовах (Calling Convention x86_64: аргументы в `rdi`, `rsi`, `rdx`, `rcx`, `r8`, `r9`, возврат в `rax`).
* Функции, предназначенные для вызова, должны быть помечены глобальными метками (например, `global encrypt_payload`).
